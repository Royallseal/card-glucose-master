# 《卡牌控糖师》文档撰写风格与规范指南

本规范旨在定义《卡牌控糖师（Card Glucose Master）》项目所有开发文档、设计案、测试指南及数值审计报告的撰写标准。通过统一术语、规范文体与格式要求，确保项目文档具备高度的专业性、一致性与可读性。

---

## 一、 核心术语与定义规范

所有文档在涉及游戏专有名词时，必须严格遵守以下术语定义，严禁在正式文档中使用任何非正式草稿名称：

### 1. 游戏命名规范
* **标准中文名称**：**《卡牌控糖师》**（书名号必须完整包覆）。
* **标准英文名称**：**Card Glucose Master**。
* **禁用词汇**：严禁使用任何开发阶段的曾用名、草稿代号或别名（例如“《血糖大作战》”、“《控糖大师》”、“控糖战役”等）。

### 2. 角色与属性术语
* **玩家角色**：**控糖师**。
* **战斗属性**：**生命值**、**格挡值**、**能量**、**血糖值**（单位统一为 `mmol/L`）。
* **血糖判定名词**：**低血糖**（检测下限）、**健康血糖**、**高血糖**（检测上限）、**高血糖危象**（血糖超限死亡）。

### 3. 状态（Buff/Debuff）词条标准
所有文档提及 Buff 和 Debuff 时必须与代码及 CSV 数据库完全对齐，书写时需加粗并保持字样统一：
* 属性修正类：**活力**（`vitality`）、**耐力**（`endurance`）。
* 状态削减类：**脆弱**（`fragility`）、**乏力**（`lethargy`）、**僵硬**（`stiffness`）。
* 被动次数盾：**缓释**（`slow_release`）、**敏化**（`sensitivity`）。

## 二、 文体风格与撰写语调

项目文档应遵循**严谨、客观、陈述性**的撰写风格。具体准则如下：

### 1. 以陈述句为主
* **要求**：使用第三人称的客观陈述句，说明事实、逻辑、设计意图和测试指令。
* **禁用词汇**：严禁使用第一人称代词（如“我”、“我们”、“咱们”），或带有口语化、引导性质的过渡句（例如“今天我们来看看...”、“其实这个地方是这样处理的...”）。
* **示例对比**：
  * *错误*：我们发现如果血糖低于 2.0 玩家就会死掉，所以咱们得加个判断。
  * *正确*：当系统检测到玩家当前的血糖值低于或等于 2.0 mmol/L 的下限临界点时，将判定战役失败并记录对应的低血糖死亡原因。

### 2. 杜绝不规范缩写与速记符号
* **要求**：所有正式文档在陈述数值逻辑时，应当书写完整中文词汇，不允许出现开发阶段的口头缩写或草稿符号。
* **示例对比**：
  * *错误*：均攻 4.5，均格 1.3，脆2，活1，TTK 3。
  * *正确*：平均攻击力为 4.5 点，平均格挡值为 1.3 点，处于脆弱状态 2 层与活力状态 1 层，预计击杀回合（TTK）为 3 回合。
  * *错误*：当将军 28 等效，脆×2 后伤害破百。
  * *正确*：当并发将军的基础单段攻击为 28 点，若玩家处于脆弱状态（伤害受击翻倍），实际承受的伤害将达到 56 点。

## 三、 排版格式与 Markdown 规范

规范的视觉排版能极大地降低跨部门沟通成本。

### 1. 结构与分级标题
* 每一个文档必须有且仅有一个一级标题（`#` 格式），用作文档主标题。
* 二级标题（`##`）必须在上下方保留一条空行并使用 `---` 分割线进行视觉分区块。
* 各级标题必须按顺序严格层递，禁止出现跳跃（例如直接从一级标题跳转至三级标题）。

### 2. 表格设计规范
* 所有数据统计、对比表均必须采用 Markdown 闭合表格。
* 表头文本必须居中或左对齐，涉及数值的一列必须居中对齐。
* 表格内部的所有名词必须拼写完整，禁止出现“乏1”、“僵2”等口头缩写。

### 3. 富文本着色与高亮
* 在文档中表达状态和键盘按键时，应当使用行内代码块包裹（例如：`UpArrow` 键，`gold` 提示 ID）。
* 涉及重要的数值临界点、按键说明或关键概念时，应当加粗显示。
* 可以使用 GitHub 风格的 Alert 信息框（如 `> [!IMPORTANT]`、`> [!NOTE]`）展示关键的设计限制。

### 4. 链接与路径规范（必须使用相对路径）
* **基本原则**：文档内部所有引用其他文档或代码源文件的链接，**必须统一使用相对路径**（例如：`[测试指南](./developer_testing_guide.md)`，而非以 `file:///` 或绝对磁盘盘符开头的绝对路径）。
* **优点**：使用相对路径可保证项目整体迁移或在不同机器、不同操作系统克隆时，Markdown 链接依然能够正常点击跳转，彻底消除绝对路径造成的死链与坏链。

### 5. 数学公式排版
* 文档中所有的数学公式，必须采用标准的 LaTeX 语法书写。
* 行内公式使用单美元符号（例如：$f(x) = ax + b$），块级公式使用双美元符号居中（例如：$$\text{公式内容}$$）。
* 公式中的所有变量必须在公式下方进行逐一文字定义，以保证非算法人员能够读懂。

### 6. 标题语言规范
* **要求**：各级标题（H1至H4）必须使用合适、精炼的中文。
* **禁用内容**：严禁在标题中附加任何不必要的英文翻译、子模块英文名称或英文后缀（例如：禁止使用 `## 一、 关卡快速跳转 (Level Jumping)`，应使用 `## 一、 关卡快速跳转`）。
* **特殊例外**：允许在标题中包含标准的英文软件/引擎名称（如 `Unity`、`Git` 等）、科学计量单位（如 `mmol/L`）、行业或项目常用英文缩写与特定概念（如 `SOP`、`Boss`、`Buff/Debuff`）以及具体的代码文件指代名（如 `BattleConstants.cs` 等）。但应保持排版简洁，避免无谓的中英文混杂翻译。

## 四、 文档质量校验清单

在提交、更新任何文档前，撰写人员必须核对以下清单：

- [ ] **游戏名称校验**：全文检索是否包含“血糖大作战”、“控糖战役”等历史错误名称。
- [ ] **第三人称检查**：全文是否已消除“我们”、“我”、“咱们”等第一人称或口语化用词。
- [ ] **表项拼写校验**：表格中所有的状态参数是否已使用完整书写形式（例如将“脆2”改为“脆弱状态 2 层”）。
- [ ] **路径链接校验**：文档中涉及的代码符号、变量名及文件路径，是否符合点击跳转的 Markdown 规范，且**全部采用相对路径**形式。
- [ ] **标题语言校验**：检查各级标题是否使用纯正中文，已清除所有不必要的英文翻译或英文子模块名称。
